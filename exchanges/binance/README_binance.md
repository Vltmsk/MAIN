# Binance Integration

Интеграция Binance для сбора 1-секундных свечей с Spot и Linear (USDⓈ-M Futures) рынков.

## 🚀 Быстрый старт

Запуск через основной модуль:
```bash
python main.py
```

Настройка в `config.py`:
```python
config.exchanges.binance_spot = True    # Spot рынок
config.exchanges.binance_linear = True  # Futures рынок
```

## 🔌 WebSocket подключение

### Реализация подключения

#### Spot подключение
- **Endpoint**: `wss://stream.binance.com:9443/stream`
- **Формат**: Мульти-стрим (несколько символов в одном соединении через URL параметры)
- **URL пример**: `wss://stream.binance.com:9443/stream?streams=btcusdt@kline_1s/ethusdt@kline_1s/ltcusdt@kline_1s`
- **Способ подключения**: Используется `aiohttp.ClientSession.ws_connect()` без параметра heartbeat
- **Ping/Pong**: aiohttp автоматически обрабатывает ping от сервера и отвечает pong
- **Лимит**: До **150 стримов** на одно подключение (безопасный запас ниже официального лимита ~200)
- **Функция**: `_ws_connection_worker()` обрабатывает соединения для Spot

#### Futures (Linear) подключение
- **Endpoint**: `wss://fstream.binance.com/ws` (подписка через JSON)
- **Формат**: Continuous Kline stream для перпетуальных контрактов
- **Способ подключения**: Используется `aiohttp.ClientSession.ws_connect()` без параметра heartbeat
- **Ping/Pong**: aiohttp автоматически обрабатывает ping от сервера и отвечает pong
- **Лимит**: До **150 стримов** на одно подключение
- **Функция**: `_ws_connection_worker_subscribe()` обрабатывает соединения для Linear

### Ping/Pong механизм

Согласно официальной документации Binance, сервер сам отправляет ping-фреймы, и клиент должен отвечать pong.

#### Spot API

**Требования от сервера Binance:**
- Сервер отправляет ping-фрейм каждые **20 секунд**
- Если сервер не получает pong в течение **1 минуты** — соединение разрывается
- При получении ping необходимо немедленно отправить pong с копией payload из ping

#### Futures (Linear) API

**Требования от сервера Binance:**
- Сервер отправляет ping-фрейм каждые **3 минуты** (180 секунд)
- Если сервер не получает pong в течение **10 минут** — соединение разрывается
- При получении ping необходимо немедленно отправить pong с копией payload из ping

**Важно:** Незапрошенные pong-фреймы разрешены, но **не предотвращают разрыв соединения**. Рекомендуется, чтобы payload для таких pong был пустым.

#### Реализация в коде

Используется библиотека **aiohttp**, которая **автоматически** обрабатывает ping/pong:

- **Автоматический ответ на ping от сервера**: aiohttp автоматически отвечает pong с тем же payload, когда получает ping от сервера Binance
- **Клиентский ping не используется**: Параметр `heartbeat` не используется, так как сервер сам отправляет ping регулярно

```python
async with _session.ws_connect(url) as ws:
    # aiohttp автоматически отвечает pong на ping от сервера Binance
    # Дополнительный клиентский ping не нужен
```

**Важно понимать**: 
- Для Spot сервер отправляет ping каждые 20 секунд — aiohttp автоматически отвечает pong
- Для Linear сервер отправляет ping каждые 3 минуты — aiohttp автоматически отвечает pong
- **Клиентский ping не используется** — нет необходимости отправлять ping от клиента, так как сервер сам отправляет ping регулярно
- aiohttp автоматически обрабатывает все ping/pong на уровне протокола WebSocket
- Согласно документации Binance, незапрошенные pong не предотвращают разрыв соединения, поэтому достаточно только отвечать на ping от сервера

### Подписка на каналы

#### Spot - Подписка через URL параметры

**Метод**: Параметры передаются прямо в URL при подключении

**Формат streams в URL**:
```
wss://stream.binance.com:9443/stream?streams=symbol1@kline_1s/symbol2@kline_1s/...
```

**Пример формирования**:
```python
spot_streams = [f"{sym.lower()}@kline_1s" for sym in spot_symbols]
url = f"{SPOT_WS_ENDPOINT}?streams={'/'.join(chunk)}"
```

**Преимущества**:
- Не требует явной подписки после подключения
- Все стримы активируются сразу при установлении соединения
- Проще в реализации

#### Linear - Подписка через JSON сообщение

**Метод**: После подключения отправляется JSON сообщение с подпиской

**Формат сообщения подписки**:
```json
{
  "method": "SUBSCRIBE",
  "params": [
    "btcusdt_perpetual@continuousKline_1s",
    "ethusdt_perpetual@continuousKline_1s"
  ],
  "id": 1
}
```

**Пример кода**:
```python
subscribe_msg = {
    "method": "SUBSCRIBE",
    "params": streams,
    "id": 1
}
await ws.send_json(subscribe_msg)
```

**Формат streams для Linear**:
- Каждый stream: `{symbol}_perpetual@continuousKline_1s`
- Пример: `btcusdt_perpetual@continuousKline_1s`

**Обработка ответа подписки**:

После отправки подписки ожидается подтверждение или данные в течение **30 секунд**:

1. **Подтверждение подписки** (JSON ответ с `id: 1`):
```json
{
  "result": null,
  "id": 1
}
```

2. **Ошибка подписки** (JSON ответ с `id: 1` и `error`):
```json
{
  "error": {
    "code": 400,
    "msg": "Invalid request"
  },
  "id": 1
}
```

3. **Данные continuous_kline** (если приходят сразу без подтверждения):
```json
{
  "e": "continuous_kline",
  "E": 123456789,
  "ps": "BTCUSDT",
  "ct": "PERPETUAL",
  "k": { ... }
}
```

**Логика обработки**:
- Если приходит подтверждение (`result: null`) → подписка подтверждена
- Если приходит ошибка → логируется и происходит переподключение
- Если приходят данные `continuous_kline` → подписка считается работающей
- Если в течение 30 секунд нет ни подтверждения, ни данных → переподключение

**Пример кода обработки**:
```python
# Проверяем ответ на подписку
if payload.get("id") == 1:
    if payload.get("result") is None:
        # Ошибка подписки
        error_msg = payload.get("error", {}).get("msg", "Unknown error")
        # Логирование и переподключение
    else:
        # Подписка подтверждена
        subscription_confirmed = True

# Если приходят данные без подтверждения
if payload.get("e") == "continuous_kline":
    # Подписка работает
    subscription_confirmed = True
    first_data_received = True
```

### Отписка от каналов

**Текущая реализация**: Явная отписка **не реализована**

**Причина**: 
- При закрытии соединения все подписки автоматически отменяются
- При остановке приложения используются `task.cancel()` для всех задач
- Не требуется явная отписка перед закрытием соединения

**Если потребуется реализовать в будущем**:
```python
# Для Spot: закрыть соединение
# Для Linear: отправить UNSUBSCRIBE
unsubscribe_msg = {
    "method": "UNSUBSCRIBE",
    "params": streams,
    "id": 2
}
await ws.send_json(unsubscribe_msg)
```

### Механизм работы

1. **Инициализация сессии**: Создаётся глобальная `aiohttp.ClientSession` для всех соединений
2. **Получение символов**: При старте загружаются все доступные торговые пары через REST API (`exchangeInfo`)
3. **Фильтрация символов**:
   - Для Spot используются пары с котируемыми активами: BTC, ETH, USDT, BNB, AUD, TUSD, BRL, GBP, USDC, TRX, EUR, BIDR, DOGE, TRY, FDUSD, AEUR
   - Для Futures используются все PERPETUAL контракты с котируемыми: USDT, USDC, BTC
4. **Разбиение на чанки**: Потоки разбиваются на чанки по 150 стримов каждый функцией `_chunk_list()`
5. **Создание соединений**: Для каждого чанка создаётся отдельная asyncio задача через `asyncio.create_task()`
6. **Параллельная обработка**: Все соединения работают параллельно через asyncio
7. **Подписка (только для Linear)**: 
   - Отправляется JSON сообщение с подпиской после подключения
   - Создаётся параллельная задача для проверки таймаута (30 секунд)
   - Если в течение таймаута нет подтверждения или данных → переподключение
8. **Реконнект**: При разрыве соединения происходит автоматический реконнект с экспоненциальным backoff

### Детали реализации реконнекта

**Алгоритм реконнекта**:
```python
reconnect_attempt += 1
delay = min(2 ** min(reconnect_attempt - 1, 5), 60)
await asyncio.sleep(delay)
```

**Параметры**:
- **Минимальная задержка**: 2 секунды (первая попытка)
- **Максимальная задержка**: 60 секунд
- **Экспоненциальный рост**: 2^(attempt-1), но не более 5 степени (максимум 32 секунды, затем 60)
- **Счётчик**: Каждый реконнект увеличивает счётчик `_stats[market]["reconnects"]`

**Обработка ошибок**:
- Все ошибки логируются через `logger.error()`
- Вызывается callback `on_error()` с деталями ошибки
- Цикл продолжается бесконечно до явной отмены задачи

## 📊 Обработка свечей

### Spot
- **Тип стрима**: `@kline_1s` (1-секундные свечи)
- **Обрабатываются**: только закрытые свечи (поле `x: true`)
- **Источник**: готовые свечи от Binance API

### Futures (Linear)
- **Тип стрима**: `continuousKline_1s` для перпетуальных контрактов
- **Формат**: `{pair}_perpetual@continuousKline_1s` (например, `btcusdt_perpetual@continuousKline_1s`)
- **Событие**: `continuous_kline` (не обычный `kline`)
- **Символ**: берётся из поля `ps` (не `s`)
- **Обрабатываются**: только закрытые свечи (поле `x: true`)

### Структура данных свечи
```json
{
  "o": "50000",  // Open (цена открытия)
  "h": "51000",  // High (максимальная цена)
  "l": "49000",  // Low (минимальная цена)
  "c": "50500",  // Close (цена закрытия)
  "q": "1000.5", // Quote Volume (объём в котируемом активе)
  "x": true      // Is Closed (флаг закрытия свечи)
}
```

## 🔄 Переподключение

Автоматическое переподключение при разрыве соединения:
- **Задержка**: экспоненциальный backoff от 2 до 60 секунд
- **Счётчик**: количество реконнектов в статистике
- **Уведомление**: логирование каждого реконнекта

## ⚙️ Лимиты и рекомендации

### Используемые лимиты

#### Лимиты в коде
- **Стримов на соединение (Spot)**: **150** (`STREAMS_PER_CONNECTION = 150`)
- **Стримов на соединение (Linear)**: **150** (`STREAMS_PER_CONNECTION = 150`)
- **Heartbeat**: Не используется — полагаемся только на автоматический ответ aiohttp на ping от сервера
- **Таймаут подтверждения подписки (Linear)**: **30 секунд** (`subscription_timeout = 30.0`)
- **Максимальная задержка реконнекта**: **60 секунд**

#### Требования Binance для ping/pong
- **Spot**: Сервер отправляет ping каждые **20 секунд**, таймаут разрыва — **1 минута** без pong
- **Futures (Linear)**: Сервер отправляет ping каждые **3 минуты (180 секунд)**, таймаут разрыва — **10 минут** без pong

#### Официальные лимиты Binance
- **Официальный лимит стримов**: ~200 на одно соединение
- **Рекомендация**: Используем 150 для безопасности (25% запас)

### Рекомендации

⚠️ **Важно соблюдать**:
- Не превышайте 150 стримов на одно соединение
- Поддерживайте стабильное интернет-соединение
- При большом количестве символов создаются несколько соединений автоматически
- Мониторьте количество реконнектов через `get_statistics()`

### Оптимизация подключений

**Автоматическое масштабирование**:
```python
# Количество соединений рассчитывается автоматически
num_connections = ceil(total_symbols / 150)
```

**Пример**:
- 250 символов Spot → 2 соединения (150 + 100)
- 180 символов Linear → 2 соединения (150 + 30)

## 📝 Особенности реализации

1. **REST API**: Получение списка символов при старте через `exchangeInfo`
2. **Мульти-стрим**: Несколько символов в одном WebSocket соединении для Spot
3. **JSON подписка**: Явная подписка через JSON для Futures
4. **Parallel processing**: Все соединения работают параллельно через asyncio
5. **Memory efficient**: Обработка готовых свечей без накопления в памяти
